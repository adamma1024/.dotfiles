# /021 — From Zero to One

A step-by-step workflow for tackling any engineering challenge from scratch. Run this command and follow each phase in order. Do NOT skip ahead — complete each phase before moving to the next.

## Phase 1: Understand & Diagnose
- Read the full project README, all source files, and any relevant docs
- Summarize: What does this system do? What are the stated problems/requirements?
- List all stakeholders and their conflicting needs
- Ask the user 3-5 clarifying questions about ambiguous requirements before proceeding
- Output a **Problem Statement** in 2-3 sentences

## Phase 2: High-Level Design
- Propose 2-3 possible architectural approaches to solve the problem
- For each approach, list: pros, cons, complexity, and which stakeholders it serves
- Identify the **big trade-offs** — what do you gain and lose with each option?
- Deep dive into the most complex or risky design area — explain why it's hard
- Ask the user: "Which direction do you want to go, and why?"
- Output a **Design Decision** with the chosen approach and reasoning

## Phase 3: Task Breakdown & Prioritization
- Break the chosen design into discrete tasks
- Classify each task: **P0** (must-have, highest impact), **P1** (important), **P2** (nice-to-have)
- For each task estimate scope: S (small), M (medium), L (large)
- Order tasks by: P0 first → P1 → P2, within each priority by impact/effort ratio
- Ask the user: "Does this priority order make sense? Anything to adjust?"
- Output a **Task Plan** as a numbered checklist

## Phase 4: TDD — Test Cases First
- For each P0 task, design test cases BEFORE writing any implementation
- Cover: happy path, edge cases, error cases, boundary conditions
- Write the actual test code (failing tests) — follow red-green-refactor
- Ask the user: "Do these test cases cover what matters? Anything missing?"
- Output: **Test files created**, all tests should be RED (failing)

## Phase 5: Code Generation — P0
- Implement P0 tasks one by one, making failing tests turn green
- After each task: run tests, confirm green, then move to next
- Keep code clean — no over-engineering, no dead code
- Ask the user to review after all P0s are done
- Output: **P0 complete** — summary of what was built and test results

## Phase 6: Code Generation — P1 & Beyond
- Repeat Phase 4-5 for P1 tasks (test first → implement → verify)
- Only proceed to P2 if time allows
- Output: **Implementation complete** — summary of all changes

## Phase 7: Manual Testing
- Start the server and test every changed endpoint/feature by hand
- Use curl, httpie, or browser — verify real behavior matches expectations
- Test edge cases: bad input, missing fields, boundary values
- Document any bugs found and fix them
- Output: **Test Report** — what was tested, what passed, what was fixed

## Phase 8: Feedback & Follow-up
- Summarize everything that was done: diagnosis → design → implementation → testing
- List known trade-offs and limitations of the current solution
- Suggest what you'd do next with more time (P2 items, optimizations, monitoring)
- Prepare talking points: be ready to explain WHY every decision was made
- Output: **Summary & Next Steps**

---

**Usage:** Run `/021` at the start of any project or challenge. The command will guide you through each phase interactively, waiting for your input at decision points.
